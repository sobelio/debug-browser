<task type="auto">
  <name>Task 1: Add props and state extraction to fiber tree walker</name>
  <files>daemon/src/scripts/get-component-tree.js, daemon/src/types.ts, daemon/src/protocol.ts</files>
  <action>
Extend the `get-component-tree.js` fiber walker to include props and state:

1. **Props extraction** from `fiber.memoizedProps`:
   - For each component fiber, read `fiber.memoizedProps`
   - Serialize props safely: handle circular refs, functions (show as `"[Function]"`), DOM elements (show as `"[Element: tagName]"`), React elements (show as `"[ReactElement: type]"`), symbols, undefined
   - Truncate string values longer than 200 chars
   - Limit object depth to 3 levels to prevent huge output
   - Filter out `children` prop (it's structural, not informative in tree view)

2. **State extraction** from `fiber.memoizedState`:
   - For class components: `fiber.memoizedState` is the state object directly — serialize it
   - For function components: `fiber.memoizedState` is a linked list of hooks. For THIS plan, extract only basic state:
     - Walk the linked list (`hook.next`)
     - For `useState`/`useReducer` hooks: `hook.memoizedState` is the current value
     - Identify hook type by checking `hook.queue` (useState/useReducer have it) vs `hook.memoizedState` being an effect object (useEffect)
     - Return array of `{ type: 'state', value: ... }` for state hooks, skip effect/memo/ref hooks (Phase 4 handles those)
   - Apply same safe serialization as props

3. **Add options to ComponentsCommand**:
   - `includeProps?: boolean` (default true)
   - `includeState?: boolean` (default true)
   - `propsDepth?: number` (max serialization depth, default 3)
   - Update protocol.ts schema accordingly

4. **Safe serialization function**: Create a `safeSerialize(value, maxDepth)` helper within the script that handles all the edge cases. This is critical — fiber objects contain tons of internal React references that will cause circular ref errors if naively serialized.
  </action>
  <verify>
Build: `cd daemon && npx tsc`. Test by navigating to a React app and running the components action, verifying props and state appear in output without serialization errors.
  </verify>
  <done>Component tree includes serialized props and state per component. No circular reference errors. Functions, DOM elements, and other non-serializable values handled gracefully.</done>
</task>

<task type="auto">
  <name>Task 2: Update CLI output formatting for props/state and add filter flags</name>
  <files>src/main.rs, src/commands.rs</files>
  <action>
Update CLI to display props and state in the component tree output:

1. **Text format** — extend the tree formatter from 03-02:
```
App (function)
  props: { theme: "dark", locale: "en" }
  Header (function)
    props: { title: "My App", showNav: true }
    NavBar (function)
      state: [{ type: "state", value: false }]
      props: { items: [Array(5)] }
```

Show props/state indented under each component, one line each. Use compact JSON-like formatting (single line if short, truncate if long). Skip empty props `{}` and empty state `[]`.

2. **CLI flags** for the `Components` command:
   - `--no-props` — hide props from output
   - `--no-state` — hide state from output
   - `--depth <N>` — max tree depth to display
   - `--component <name>` — filter to show only components matching name (substring match)
   Pass these as fields in the daemon command JSON.

3. **commands.rs**: Update `components()` to accept optional parameters and include them in the JSON command.

4. **Graceful non-React handling**: If the daemon returns `{ roots: [], componentCount: 0 }` or `detected: false`, print a helpful message: "No React components detected on this page. Make sure you navigate to a React application first."
