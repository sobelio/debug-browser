**State save command:**
- Daemon: Add `state_save` action with `path: string` parameter
- Handler: Call `page.context().storageState({ path: command.path })` — saves cookies + localStorage to JSON file
- CLI: `state save <path>` subcommand

**State load (--state flag at launch):**
- This is NOT a runtime command — storage state must be loaded at context creation time
- Add optional `storageState?: string` to LaunchCommand in types.ts
- In browser.ts launch(): If storageState provided, pass to `browser.newContext({ storageState: options.storageState })`
- CLI: Add `--state <path>` global flag (like --connect). Pass to launch command.
- If user tries `state load` as a command, return informational error: "Storage state must be loaded at launch. Use: debug-browser --state <path> navigate <url>"

**Validation:**
- state save: Verify path is writable (let Playwright handle errors)
- --state: Verify file exists before passing to daemon

**Output:** state save → `Saved to <path>`. --state at launch → transparent (no extra output).

<task type="auto">
  <name>Task 2: Add --profile flag for persistent browser context</name>
  <files>daemon/src/types.ts, daemon/src/protocol.ts, daemon/src/browser.ts, src/main.rs</files>
  <action>
Add `--profile <path>` flag for persistent browser contexts that survive restarts.

**Daemon types.ts:** Add optional `profile?: string` to LaunchCommand interface.

**Daemon protocol.ts:** Add `profile: z.string().optional()` to launch schema.

**Daemon browser.ts launch():**
- If `options.profile` provided:
  - Expand `~` to `os.homedir()` (not shell-expanded in daemon)
  - Use `chromium.launchPersistentContext(profilePath, { headless, viewport, executablePath })` instead of `chromium.launch()`
  - This returns a BrowserContext directly (no separate Browser instance)
  - Store the context and get page from it: `context.pages()[0] || await context.newPage()`
  - Set a flag `isPersistentContext = true` for cleanup handling
- If `options.profile` AND `options.storageState` both set → error: "Cannot use --profile with --state (profile already persists state)"
- If `options.profile` AND `options.cdpUrl/cdpPort` → error: "Cannot use --profile with --connect"

**Rust CLI (main.rs):**
- Add `--profile <path>` global flag alongside --session, --connect, --state
- Pass to launch command JSON

**Validation:**
- Profile + state = error
- Profile + connect = error
- Profile directory auto-created by Playwright

**Output:** No special output. Profile silently persists cookies/cache/storage between sessions.
